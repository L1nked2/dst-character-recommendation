# 추천 기준 v2: 4팩터 정규화 모델

## 결론

4개의 이진 축으로 16가지 기본 유형을 만듭니다. Wonkey는 이 16가지 유형에 넣지 않고, 17번째 와일드카드 유형 `WNKY`로 별도 처리합니다. 결과 화면에서는 Wonkey도 다른 캐릭터와 같은 형식으로 보여줍니다.

추천 단위는 `캐릭터`입니다. 세부 플레이 방향은 후보로 만들지 않습니다. 캐릭터 안에서 플레이 방향이 갈리더라도 대표 플레이 방향 하나로 평가합니다.

```text
추천 후보 = 캐릭터 + 대표 플레이 방향
```

## 4개 정규화 축

각 축은 `-1`부터 `+1`까지의 연속값으로 저장합니다. 유형 코드는 사용자에게 직접 보여주지 않고, URL path/state/debug 용도로만 사용합니다.

| 축 | 코드 | -1 방향 | +1 방향 | 설명 |
| --- | --- | --- | --- | --- |
| 활동 위치 | `O/B` | `O`: Outfield | `B`: Base | 밖에서 탐험/채집/전투를 하는지, 기지/농장/장치를 구축하는지 |
| 플레이 방식 | `D/S` | `D`: Direct | `S`: Setup | 직접 조작/전투로 푸는지, 세팅/소환/장치/루프로 푸는지 |
| 효과 대상 | `S/T` | `S`: Self | `T`: Team | 본인이 직접 성과를 내는지, 팀/인프라가 성과를 내게 하는지 |
| 요구 난도 | `E/H` | `E`: Easy | `H`: Hard | 실수 허용도, 운영 복잡도, 숙련 보상 |

16가지 기본 유형은 이 4축의 조합입니다. 단, 이 코드는 내부 식별자입니다.

내부 코드 예:

- `ODSE`: 야외에서 직접 해결하는 저마찰 자기완결형
- `ODSH`: 야외에서 직접 해결하는 고숙련 자기완결형
- `OSTE`: 야외에서 세팅을 활용해 팀을 돕는 저마찰 지원형
- `BSTH`: 기지 기반 세팅을 설계하는 고숙련 팀 지원형

와일드카드:

- `WNKY`: Wonkey 전용 특수 결과. 4축 점수로 직접 경쟁시키지 않고, 특정 응답 패턴에서 캐릭터 결과처럼 노출합니다.

## 축별 의미

### 활동 위치: `O/B`

`O`는 맵 바깥에서 가치가 나는 성향입니다.

- 탐험
- 폐허/동굴/바다 원정
- 사냥과 보스전
- 기본 자원 채집
- 위험 지역 진입

`B`는 거점과 장기 운영에서 가치가 나는 성향입니다.

- 기지 방어
- 농장과 식량 체계
- 장치와 자동화
- 세력 구축
- 공용 자원 생산

### 플레이 방식: `D/S`

`D`는 직접 조작으로 문제를 해결하는 쪽입니다.

- 카이팅
- 원거리 조준
- 빠른 이동
- 위험한 타이밍 진입
- 직접 채집

`S`는 사전 세팅이나 특수 규칙으로 문제를 해결하는 쪽입니다.

- 소환수
- 장치
- 책/마법
- 농장 루프
- 캐릭터 전용 자원
- 반복 작업 자동화

### 효과 대상: `S/T`

`S`는 본인이 직접 생존, 전투, 채집 성과를 내는 쪽입니다.

- 솔로 적합
- 직접 전투력
- 자기 회복/기동성
- 자기완결 루프

`T`는 팀이나 인프라가 더 잘 굴러가도록 만드는 쪽입니다.

- 힐
- 버프
- 장비 공급
- 음식 공급
- 기지 방어
- 동선/이동 지원

### 요구 난도: `E/H`

이 축은 "캐릭터가 강한가"를 직접 평가하지 않습니다. 대신 사용자가 체감하는 실효 난도를 측정합니다.

`E`는 다음 성질을 가집니다.

- 실수 복구가 쉽다.
- 기본 루프가 단순하다.
- 준비 의존도가 낮다.
- 초반부터 가치가 나온다.

`H`는 다음 성질을 가집니다.

- 운영할 자원이 많다.
- 후반 조건 의존도가 있다.
- 숙련 시 보상이 크다.
- 실수 비용이 크다.

## 점수 계산 방식

사용자 응답과 후보 캐릭터는 같은 4차원 벡터를 가집니다.

```js
profile = {
  locus: -1..1,      // O/B
  style: -1..1,      // D/S
  target: -1..1,     // S/T
  difficulty: -1..1  // E/H
}
```

후보 점수는 가중 거리 방식으로 계산합니다.

```text
distance = sum(axisWeight * abs(userAxis - candidateAxis))
score = 100 - normalize(distance)
```

기본 축 가중치:

| 축 | 가중치 | 이유 |
| --- | --- | --- |
| 활동 위치 | 1.0 | 플레이 루틴을 크게 바꿈 |
| 플레이 방식 | 1.0 | 손맛과 운영감을 크게 바꿈 |
| 효과 대상 | 0.9 | 솔로/팀 선호 반영 |
| 요구 난도 | 1.2 | 추천 실패 비용이 가장 큼 |

## 후보군 모델

```text
CharacterCandidate {
  id: string
  characterName: string
  unlock: "free" | "dlc" | "wildcard"
  representativePlaystyle: string
  tags: string[]
  typeCode: "ODSE" | "ODSH" | ... | "WNKY" // internal only
  profile: {
    locus: number
    style: number
    target: number
    difficulty: number
  }
  summary: string
  recommendedFor: string[]
  caution: string
}
```

결과 화면은 캐릭터와 대표 플레이 방향을 보여줍니다. 유형 코드는 표시하지 않습니다.

```text
추천: Woodie
#탐험 #채집 #안정형
대표 방향: 채집/정찰 중심 야외 노동
이유: 야외 탐험, 직접 조작, 안정적인 생존 선호와 일치
주의: 전투 캐리보다 정찰/자원 가치가 큼
```

결과 화면에는 조합 정보도 함께 표시합니다.

```text
잘 어울리는 캐릭터
- Winona
- Warly
```

조합 정보는 [캐릭터 후보 v2](character-map-v2.md)의 조합 매트릭스를 기준으로 합니다.

## 후보 수 관리

| 그룹 | 후보 수 |
| --- | --- |
| 일반 캐릭터 | 18 |
| Wonkey 와일드카드 | 1 |
| 총 후보 | 19 |

결과 화면은 다음만 보여줍니다.

- 1순위 추천 캐릭터
- 2~3개 대안 캐릭터
- 잘 어울리는 캐릭터
- Wonkey 조건 충족 시 와일드카드 결과

## 동점 처리

거리 점수가 같거나 매우 비슷하면 초보/Easy 성향에 더 안전한 후보를 우선합니다.

동점 처리 순서:

1. 사용자 `difficulty`가 Easy 방향이면 더 쉬운 캐릭터를 우선합니다.
2. 사용자 `difficulty`가 Hard 방향이면 거리 점수가 더 가까운 캐릭터를 그대로 우선합니다.
3. 그래도 같으면 `target` 축이 더 가까운 캐릭터를 우선합니다.
4. 그래도 같으면 기존 정렬 순서를 유지합니다.

## 구현 순서

1. 현재 `traits` 기반 캐릭터 데이터를 `characterCandidates`로 교체합니다.
2. [질문지](questions.md)의 12문항과 [질문-팩터 매핑 v2](question-factor-map-v2.md)를 기준으로 질문 데이터를 4축 벡터로 교체합니다.
3. [캐릭터 후보 v2](character-map-v2.md)를 기준으로 캐릭터 벡터를 적용합니다.
4. 결과 화면에 `캐릭터명 + 대표 플레이 방향`을 표시합니다.
5. 결과 화면에 잘 어울리는 캐릭터를 표시합니다.

## URL/State 사용

유형 코드는 공유 URL이나 내부 상태에만 사용합니다.

예:

```text
/result?profile=odse&character=woodie
/result?profile=wnky&character=wonkey
```

UI 표시 문구는 코드가 아니라 자연어를 사용합니다.

```text
야외에서 직접 움직이며 안정적으로 자원을 확보하는 타입입니다.
```

## Wonkey Rule

```text
Wonkey는 일반 거리 기반 추천에는 포함하지 않는다.
사용자 응답이 변칙/제약/탐험/고난도 방향으로 강하게 쏠릴 때
결과 화면에서 캐릭터 결과 형식으로 노출한다.
```
